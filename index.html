Android Training 之Service简单用法

Posted on 2016-08-26

Service是Android四大组件之一，在平时的开发中占据着重大的地位。所谓Service，即服务，不需要和用户进行界面的可以在后台长时间运行的一种组件。从定义可以看出，Service主要用来处理后台耗时操作，比如网络，音乐播放，处理I/O等，并且在应用无前台焦点情况下也可存活进行任务处理。

Service启动方式

Service的启动主要有两种方式

  1. 启动Service：该方式启动的Service可以独立存在，并且理想情况下可以永远存活。该方式主要是通过startService(Intent intent)方法来启动，启动后，会回调Service的onStartCommand()方法。
  2. 绑定Service：该方式主要是将Service与其他组件进行绑定，这样其他组件就可以和Service进行通信，比如发送请求，获取结果等等。该方式主要是通过bindService() 方法进行绑定，会回调onBind()方法。一个Service可以与多个组件进行绑定，当所有绑定取消之后，该服务才会销毁。

Service是一个抽象类，因此要使用服务必须得自定义Service并且继承Android的抽象Service类，实现onBind()方法。对于启动Service方式，可以覆写onCreate()方法和onStartCommand()方法。onCreate()方法只会在Service第一次启动时回调，一旦Service已经存活，则不再进行回调，因此onCreate()方法中可以做一些初始化操作。而每次通过startService()方法启动Service都会回调onStartCommand()方法，因此可以在onStartCommand()方法中执行操作任务。值得一提的是，Service并不会在新的线程中执行生命周期，其所在线程和宿主线程即启动该Service的组件的线程相同，因此，如果要在Service中做耗时操作需要另起线程。如果需要停止Service，只需要在Service中调用stopSelf()方法或者在其他组件中调用stopService()方法即可。该方式比较简单，就不贴代码了。

和Activity一样，Service也需要在AndroidManifest文件中进行注册，注册方式和Activity相似。此外还可以为Service添加exported属性，该属性可以控制外部应用对Service的启动效果。

关于onStartCommand()方法的返回值。该方法返回一个整形数字，主要有一下几种返回值。

  1. START_NOT_STICKY:如果系统在 onStartCommand() 返回后终止服务，则除非有挂起 Intent 要传递，否则系统不会重建服务。这是最安全的选项，可以避免在不必要时以及应用能够轻松重启所有未完成的作业时运行服务。
  2. START_STICKY:如果系统在 onStartCommand() 返回后终止服务，则会重建服务并调用 onStartCommand()，但绝对不会重新传递最后一个 Intent。相反，除非有挂起 Intent 要启动服务（在这种情况下，将传递这些 Intent ），否则系统会通过空 Intent 调用 onStartCommand()。这适用于不执行命令、但无限期运行并等待作业的媒体播放器（或类似服务）。
  3. START_REDELIVER_INTENT:如果系统在 onStartCommand() 返回后终止服务，则会重建服务，并通过传递给服务的最后一个 Intent 调用 onStartCommand()。任何挂起 Intent 均依次传递。这适用于主动执行应该立即恢复的作业（例如下载文件）的服务。

对于绑定式Service，过程稍微复杂一点。先看一下代码。

public class LaunchService extends Service {
  private LocalBinder localBinder = new LocalBinder();

  public class LocalBinder extends Binder {
    LaunchService getService() {
      return LaunchService.this;
    }
  }

  @Nullable
  @Override
  public IBinder onBind(Intent intent) {
    return localBinder;
  }


  @Override
  public void onCreate() {
    super.onCreate();
  }

  @Override
  public int onStartCommand(Intent intent, int flags, int startId) {
    return super.onStartCommand(intent, flags, startId);
  }

  public void hello() {
    System.out.println("hello");
  }
}

在自定义Service中，我们需要提供一个继承自Binder类，在该类中写一个获取Service实例的方法。然后在onBind()方法中返回自定义的Binder实例。这样在绑定该Service的组件中就可以通过获取到自定义Binder实例，并且通过Binder实例获取到Service实例，这样就形成了绑定。在Activity中实现绑定的代码如下所示。

private LaunchService launchService;

private ServiceConnection serviceConnection = new ServiceConnection() {
    @Override
    public void onServiceConnected(ComponentName componentName, IBinder iBinder) {
      launchService = ((LaunchService.LocalBinder) iBinder).getService();
      launchService.hello();
    }

    @Override
    public void onServiceDisconnected(ComponentName componentName) {

    }
  };

@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_main);
    Intent intent = new Intent();
    intent.setClass(this, LaunchService.class);
    bindService(intent, serviceConnection, BIND_AUTO_CREATE);
}

bindService()方法需要传入三个参数，第一个参数为Intent，在Intent中申明需要绑定的Service，第二个参数为ServiceConnection，该参数中实现绑定完成以及解除绑定的回调，第三个参数为整形的flag，该值和内存管理以及调试等相关。一般情况设置为BIND_AUTO_CREATE即可。值得一提的是，绑定过程是异步的，因此调用bindService之后并不能保证立即获得Service实例。此外，我们应当在合适的时间取消服务绑定，如Activity暂停或者销毁时调用unbindService()方法。

由于Service启动和绑定可以同时存在，为了避免绑定Service方法调用onCreate或者onStartCommand方法，我们可以在Binder中返回接口类型实例。同时，一个Service实现不同的接口，则可以通过不同的Binder类返回不同的接口，这样可以实现一个Service实现不同的绑定。

Service生命周期

如同Activity一样，Service也有生命周期，如下图所示

两种方式的生命周期类似，只有第一次启动Service时会回调onCreate方法，每次启动或者绑定Service时，回调onStartCommand()方法或者onBind()方法。停止服务或者解绑服务时，都会回调onDestroy()方法，不同的是绑定式方式在解绑时还会回调unBind()方法。

前台Service

从意义上来说，Service一般负责后台服务，然而也有一些情况是需要前台服务的，这将比后台服务生命力更强，系统一般无法自动回收前台服务。要使用前台Service，只需要调用startForeground()方法即可，该方法接受一个整形id和Notification实例。

实际开发中，关于Service主要遇到的问题是线程问题以及Service存活问题。关于线程，只需要记住Service和宿主处于同一线程原则即可，而Service生命周期问题，得记住Service的生命周期图以及Service永存的简单方法，即通过onStartCommand方法的返回值控制Service的生命。

Android Training之数据保存

Posted on 2016-08-01

在Android系统中，保存数据主要有三种方式：

  1. 轻量级数据保存方案：SharedPreferences，主要以键值对的方式保存，一般用来保存应用的设置，用户等轻量级信息。
  2. 文件保存方案：重量级数据可以通过文件的方式存储到系统的内置存储区域或者外置存储区域。
  3. 结构性数据保存方案：对于应用的结构化数据，可以使用数据库的方式来进行存储，在Android中主要使用SQLite来进行数据库存储。
